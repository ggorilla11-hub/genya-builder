## 0. 현황 먼저 (파악 결과 · 정직)

- 메인 UI = `deploy/genya.html`(283KB SPA, 로그인/온보딩/메인 한 파일). 메인=view 5.
- **우측 상단(topbar)엔 연결·명단 버튼이 없음** — 지금은 다운로드·공유·발송기설치 아이콘 3개뿐. (회장님 지적 정확)
- 커넥터 UI는 **우측 컬럼 패널**(genya.html:703~)에 4카드(캘린더·시트·드라이브·Gmail). 상태는 `/api/conn/status`가 **각 API를 실제 호출**해 정확 판정(추측 아님·신뢰 가능).
- **재업로드 경로는 사실 "있긴" 있음**(만기 배너 `rosterHint`·KPI [지금 올리기]) — 그런데 **만기 기능에 묻혀 있어 회장님이 못 찾음.** → 문제는 "없다"가 아니라 **"안 보인다".**

### ⚠️ 더 큰 발견 (설계에 반영)
현재 **명단 업로드는 "분석 후 폐기"라 회원 시트에 저장되지 않습니다**(`/api/coverage/analyze`·`/api/manage/dashboard`로 base64 분석만). 그런데 **Step 2-B(CRUD)·2-C(결재함)는 `지니야빌더_데모_명단` 시트를 대상으로 동작**합니다.
→ **지금은 "올린 명단"과 "지니야가 쓰는 명단"이 끊겨 있습니다.** 이걸 잇는 게 이번 UX의 핵심 가치.

## 2. 커넥터 상태 표시 · A/B/C

| 안 | 방식 | 장점 | 단점 |
|---|---|---|---|
| **A ⭐** | **실측**(`/api/conn/status` 실제 probe) + 명확 시각화(🟢연결됨/⚪미연결·[재연결]) + 드라이브 파일수 | 이미 정확(거짓말 안 함) · 회장님 "불안" 해소 | 4개 API 호출(1~2초) |
| B | 스코프 문자열 추측 | 빠름 | **부정확**(이미 "거짓말 금지"로 폐기된 방식) |
| C | 캐시된 실측 + 수동 새로고침 | 빠름+정확 | 캐시 신선도 관리 |

**추천 ⭐ = A** — 신뢰가 핵심. 실측 그대로 쓰되 **드로어에서 크게·명확히** 보여주고 [🔄새로고침]·[재연결] 원클릭. 드라이브는 최근 파일수(가능하면 용량)도.

## 3. 명단 업로드 · A/B/C

| 안 | 방식 | 장점 | 단점 |
|---|---|---|---|
| A | 지금처럼 분석만(폐기) · 노출만 강화 | 변경 작음 | **명단이 시트에 안 남음** → CRUD·결재함과 계속 단절 |
| **B ⭐** | 업로드 → **회원 본인 명단 시트로 저장**(교체/추가 선택) | 올린 명단이 **진짜 지니야 명단** → CRUD·결재함·만기 전부 그걸 사용 · 부트캠프 필수 | 파싱→시트쓰기 구현(스코프는 이미 있음) |
| C | 분석 미리보기 + 저장 선택 | 유연 | 2단계라 약간 복잡 |

**추천 ⭐ = B** — 이게 이번 태스크의 진짜 임팩트. 업로드가 **회원 구글시트 명단으로 저장**돼야 지니야가 매일 그 명단으로 일합니다(제로인그레스 유지: 서버 RAM 파싱→회원 시트 write→폐기, 서버 저장0).
- 안전: 교체 시 **기존 명단 백업 안내**·미리보기·확인. 컬럼 자동 매핑(Step 2-B 동의어 재사용).

## 4. 구현 계획 (엄마1 무접촉·라우터 무접촉)

- **UI:** genya.html topbar에 버튼 1개 + 드로어(모달) **추가**. 기존 뷰·라우팅 무수정.
