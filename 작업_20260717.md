## 작업 3 — 다운로드함(MEM 연동) : 규칙③으로 멈춤·보고 (코드 미작성)
조사만 하고 코드는 짜지 않음. 이유 = 전제와 코드가 어긋나 추측 없이 진행 불가.

**코드 실측으로 발견한 사실:**
1. 저장처: 증권분석·연금 요약은 `genya_mem` 컬렉션(`genya_mem_module.js`, `main_server.js:605·618`이 저장). 조회는 `/api/mem/search`(main_server.js:661). 반환 필드 = `{userId,date,고객명,skill,summary,담보금액,timestamp}`.
2. **삭제 라우트 불일치(중요):** 대표님 지시의 `/api/memory/delete`(main_server.js:758)는 **다른 모듈(`memory_module`, `memory` 컬렉션)**용. `genya_mem` 삭제 라우트는 **없음**. → 다운로드함 [삭제]가 붙을 서버 엔드포인트가 없다. (대표님 "기존으로 되는지 먼저 확인" → 확인 결과: 안 맞음)
3. **상품비교 미저장:** `/api/compare`(586)는 `_memSaveDesign` 미호출 → MEM에 안 쌓임. 다운로드함에 상품비교를 띄우려면 서버 저장 배선 추가 필요.
4. 실증 불가: "증권분석 1건→다운로드함 1건→삭제" 1차 통과는 로그인+실증권+MEM 데이터 필요 → 무인 불가.

**대표님 결정 필요(서버 수정 승인):**
- (A) `genya_mem` 삭제용 라우트 `GET /api/mem/delete?id=` 신설할지 (Firestore document delete)
- (B) `/api/compare`에도 `_memSaveDesign` 추가해 상품비교를 다운로드함에 등록할지
- (C) 실증용 대표님 로그인 세션 + 증권 샘플

→ (A)(B)는 서버 파일 수정이 필요. 대표님이 작업2에서 "서버 손대지 마라" 하셨기에, 서버 수정은 **승인 후** 진행. 규칙③에 따라 여기서 멈추고 보고. 작업4~8은 규칙 "다음으로 넘어가지 마라"에 따라 대기.
