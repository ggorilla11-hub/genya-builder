# 지니야빌더 PHASE 5b — 데이터 격리 설계 (확정본)

> 작성: 2026-06-20 (엔진=엄마 / 코치 검토·승인). **구현 전 설계 문서.**
> 대원칙: **읽기 격리 + tenant 미들웨어만.** 발행은 OWNER 전용 유지(발행경로 0접촉). 성급한 멀티테넌트 금지 — 파일럿 1~2명 누수 검증 후 점진.
> ★★ **아래 15문항 = B3 합격 기준. 하나라도 깨지면 출시 중단.**

---

## 0. 출발점 (엔진 현실, 정직)

현재 엔진의 **모든 주요 데이터가 단일 테넌트(대표 오상열)** 다. tenant 개념은 `ACCOUNTS`(5a 산물)뿐.

| 저장소 | 내용 | 현재 소유 |
|---|---|---|
| HISTORY·DIARY·NOTIFY·PENDING | 대화·영업일기·알림·승인대기 | 대표 |
| YTLEADS·LEADS | 유튜브·네이버 리드 | 대표 |
| YTPUB·IGPUB·PROMO | 발행대장·홍보 | 대표 |
| PAYSEEN·결제시트 | 매출 | 대표 |
| genyaProfile | 직업·KPI 정체성 | 대표 |
| **ACCOUNTS** | email→tenant_id | 유일한 tenant-aware (5a) |

⚠️ **핵심 위험:** B1 읽기 엔드포인트(`/dashboard/all`·`/diary`·`/ytleads/today`·`/campaign/stats` 등)는 현재 **무인증 공개**. 로그인한 교육생이 호출하면 **대표 데이터가 보임 = 누수.** → 5b가 막는다.

---

## 1. 경계 — per-tenant / 공용 / OWNER 전용

- 🔴 **per-tenant (격리 필수):** 영업일기·리드·대화기록·알림·승인대기·발행대장·매출/결제·프로필(직업·KPI)·본인 채널 OAuth 토큰
- 🟢 **공용 (shared):** 엔진 코드·붕어빵 틀·안전 로직·capability 플래그·RAG 금융지식·INFRA/복제 템플릿·LIMIT_GUIDE
- 🔵 **OWNER 전용 (대표만, 교육생 0 노출):** 현재의 모든 글로벌 데이터(대표's YTLEADS/DIARY/결제 등) = **OWNER 테넌트로 귀속.** 교육생 = 새 빈 테넌트.

> **5b = "누수 0" 안전망**이지 풀 멀티테넌트 빌드가 아니다. 교육생별 데이터 파이프라인 풀빌드(채널 OAuth·리드수집·발행)는 그 다음 멀티테넌트 본과제(`multiJob15`).

---

## 2. 격리 미들웨어 — 세션 tenant_id 강제 스코핑

**원칙: 핸들러는 tenant를 파라미터로 받지 않는다.** (URL·쿼리·바디로 tenant 주입 불가 = 위조 차단.) 오직 **서명된 세션 쿠키의 tenant**만 신뢰.

```
tenantMiddleware(req): req.tenant = readSession(req)?.tenant ?? null   // HMAC 서명검증된 것만
isOwner(req):          req.tenant === OWNER_TENANT (OWNER_EMAIL env로 식별)
```

- 데이터 접근은 `getDiary(req.tenant)`·`getLeads(req.tenant)` 식 **tenant 인자 필수** 함수로 통과 → tenant 없으면 빈/거부.
- 글로벌(대표) 데이터는 `isOwner`일 때만 실데이터, 그 외엔 tenant-scoped 빈/본인 뷰.
- 교육생 store는 초기 비어있음 → 교육생은 **본인 빈 데이터**(대표 것 0). (데모 아님 — 실제 본인 빈 상태)

---

## 3. 확정된 3개 결정

**① B1 공개읽기 = 세션 필수 전환 (권장 A 확정)**
- 비로그인 = 빈 뷰(/401). OWNER 세션 = 기존 전체 데이터 정상(**회귀 0**). 교육생 세션 = 본인 것만.
- 결과: 시뮬 `?engine=`만으론 이제 데이터 0 → **OWNER가 로그인해야 보임 = 격리 정상.** (대표도 로그인)

**② OWNER 식별 = `OWNER_EMAIL` env (대표 이메일)**
- 코드 하드코딩 금지. 그 세션의 tenant만 기존 글로벌 데이터 접근.

**③ per-tenant 저장 = 2단계**
- 파일럿: **시트 탭 prefix `{tenant}_`** (예: `{tenant}_영업일기`) — 빠른 누수 검증.
- 부트캠프 확대 전: 탭 폭증·구글시트 한계 → **테넌트별 스프레드시트 또는 본인 Drive**로 강화 (B4 원칙과 연결).

## 4. 발행·자동발송 0접촉 유지선 (발행 PROTECT)

- 5b는 **읽기 격리 + tenant 미들웨어**만. 발행 파이프라인(60초 시계·발행함수·자동발송)은 **OWNER 테넌트 전용 유지**.
- 미들웨어는 발행 트리거 경로에 안 닿음 → `setInterval`·`solapi.send`·발행함수 **0접촉**.
- 교육생은 발행 capability 자체가 없음(채널 미연결) → **구조적으로 발행 불가**.

## 5. ★★ 데이터 격리 15문항 체크리스트 = B3 합격 기준

**계정:** A=교육생A(테스트1), B=교육생B(테스트2), OWNER=대표. **합격선 = 15문항 전부 O. 하나라도 X면 출시 중단.**

| # | 시나리오 | 합격 기준 |
|---|---|---|
| 1 | 비로그인 `/dashboard/all` | 대표 데이터 0 (빈/401) |
| 2 | A세션 `/dashboard/all` | A 것만 (대표·B 0) |
| 3 | A세션 `/diary` | A 영업일기만 |
| 4 | A세션 `/ytleads/today` | A 리드만 (대표 핫리드 0) |
| 5 | A세션 `/campaign/stats` | A 매출만 (대표 매출 0) |
| 6 | A세션 `/notify` | A 알림만 |
| 7 | A세션 `/history` | A 대화기록만 |

| 9 | **쿠키 위조**(B tenant 손삽입 가짜 쿠키) | HMAC 서명 불일치 → 거부 |
| 10 | **쿼리 주입** `?tenant=B` | 무시 (세션만 신뢰) |

## 6. 5b 착수 순서 (구현 시 단계 — 각 단계 롤백태그→가산→announce→승인→실측)

- **5b-0:** 롤백 태그
- **5b-1:** `tenantMiddleware` + `OWNER_EMAIL` 판정 (비-OWNER = 빈 뷰, OWNER = 기존대로)
- **5b-2:** 데이터 접근을 tenant 인자 필수 함수로 통과 (핸들러 tenant 무파라미터)
- **5b-3:** per-tenant 저장소 골격 (시트 탭 prefix `{tenant}_`)
- **5b-4:** ★★ 15문항 누수 테스트 (2계정) → **전부 O 후에만 다음**
- 발행·자동발송 0접촉 유지

---

## 7. B4 연결 (확대 전 강화)

부트캠프 확대 전, per-tenant 저장을 **테넌트별 스프레드시트 또는 교육생 본인 Google Drive**로 옮긴다 (제로 인그레스 안전선의 최종형 = 데이터가 교육생 본인 계정에만). 5b 파일럿(탭 prefix)에서 격리 로직 검증 → B4에서 물리 격리 강화.
