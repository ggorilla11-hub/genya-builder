# 02. 핵심 구조 — 지니야 한 제품, 총괄 모드(제니야) 통합

## 가장 중요한 결정

제니야와 지니야는 별개의 두 시스템이 아니다. 같은 구조다.

- 제니야를 따로 만들지 않는다. 지니야 하나에 "총괄 모드"를 넣는다.
- 제니야 = "총괄 모드가 켜진 지니야"이자, 오상열 대표 전용(모든 지니야를 총괄하는 최상위).
- 비유: 별도 제품이 아니라, 같은 차에 "총괄 엔진" 옵션을 켜면 제니야급이 된다.

## 붕어빵 틀 원리 (맞춤화)

모든 지니야는 같은 틀을 쓰고, 고객마다 3가지만 갈아 끼운다.

1. 정체성(시스템 프롬프트): 직업·목표·말투
2. 지식(RAG): 그 업종 자료
3. 손(도구): 그 직업에 필요한 기능

틀은 같고 "정체성·지식·손"만 바꾼다.

## 프로젝트 × 에이전트 교차 구조 (총괄 모드 핵심 UI)

- 상단 = 프로젝트(부트캠프 / 일반인 비대면강의 / 전문가 대면강의 / 일반인 상담 / AI머니야 일반 / AI머니야 전문가 + 추가)
- 좌측 = 에이전트 군단(발굴 / 관리 / 마케팅 / 디자인 / 개발 / 법무·보안 / 매출·재무 + 총괄)
- 교차: "부트캠프" 켜고 "발굴" 누르면 발굴 에이전트가 부트캠프 맥락으로 일한다.
- 효과: 에이전트 N명 × 프로젝트 M개 = N×M 가지 일. 군단이 프로젝트만 바꿔 입는다.
- 우측 = 도구 세트(구글 4종: 캘린더·이메일·드라이브·시트 / OCR / 스킬·MCP / 상급: 컴퓨터유스·자율루프는 자물쇠).
- 중앙 = 밤샘 결과물 + 자연어 대화 + 업로드(고객자료·증권·문서) + 음성/텍스트 입력.
- 이 교차 구조는 프로젝트가 여럿인 풀옵션(제니야)용. 행정사 등은 기본 단순 홈으로 둔다.

## 멀티 에이전트 회의 (24시간 핵심 목표)

- 대표가 주제를 주면 → 제니야(총괄)가 7명에게 각자 역할로 질문 → 각 에이전트가 자기 관점으로 답 → 제니야가 취합·정리 → 대표 보고.
- 대표가 끼어들어 특정 에이전트와 다시 대화 가능. 회의 종료도 대표가.
- 호명 라우팅: "발굴팀장, 내일까지 10명 발굴해줘"처럼 부르면 그 에이전트가 응답.

## 빌더가 고객에게 제공되는 형태

- 교육생이 받는 것 = 웹 주소 하나. 핸드폰에서 열고 구글 로그인. 설치 없음.
- 본사가 운영하는 하나의 빌더 사이트, 로그인하면 각자 공간(유튜브식).
- 흐름: 주소 → 구글 로그인 → 온보딩(이름·직업·고민·목표) → 기본 홈 자동 생성(1기부터 자동) → 총괄과 대화로 맞춤 구축.
- 맞춤 기능: 1기는 수업 중 함께 구축(반자동) → 2기부터 자동화.

## 개발 요청 처리 흐름

- 코드 개발이 필요 없는 요청(서류·발굴·분석 등)은 지니야가 바로 실행.
- 새 기능/화면이 필요한 요청은: 1기엔 본사의 개발 에이전트(엄마손)가 만들어 그 사용자 공간에 배포. 나중엔 샌드박스+승인 범위에서 자동 개발로 키운다. 인프라가 본사 운영이라 고객이 직접 코드 배포는 안 한다.

## 직업별 사전 인터뷰 → 직업별 템플릿

- 빌더가 "직업만 들어도 척척" 만들려면 그 직업 니즈를 미리 알아야 한다. 대표의 25년·2천 상담 통찰이 재료.
- 직업별 페인포인트 사전 인터뷰 → 직업별 기본 기능 템플릿 사전 구축. 도장깨기(1:1 제작)가 곧 인터뷰.

## 명명 규칙

- 인프라(Vercel/Render) = jenya
- 대표 총괄 = zenya(제니야)
- 고객 기본 = genya(지니야)
