# 단체카톡 real화 — 설계 검토안 v2 (나인톡 모델, 안전 최우선)

> 작성: 2026-06-21 개정 (엔진=엄마 / 코치 검토·승인 대기). **구현 전 설계 문서 — 검토안.**
> ★★★ **1번 원칙 (설계 최상위):** **"회원 한 명의 카톡 계정도 잃지 않는다." 안전이 기능보다 우선.** — 안전과 기능이 충돌하면 무조건 안전.
> 대표 결정: 방식 = **나인톡 모델**(본인 카톡 개인톡 순차발송·본인 1클릭). 발행 PROTECT·회귀 0.

---

## 0. ★먼저 짚어야 할 2가지 (구현 전 반드시 — 코치 "막히면 먼저 알려줘")

### 0-A. 아키텍처 현실 — **클라우드 엔진(jenya)은 본인 카톡을 못 보낸다**
- 나인톡 모델 = **본인 PC의 카카오톡 데스크톱 앱을 사람처럼 조작**(창 열기·붙여넣기·전송). 이건 **윈도우 데스크톱 GUI 자동화**다.
- jenya 서버는 **Render(리눅스 클라우드)** — 카톡 PC앱이 없고, 대표 화면을 조작할 수 없다. **클라우드만으로는 한 건도 못 보낸다.**
- AI손(Playwright)도 **웹 브라우저**용이라 카톡 PC앱(네이티브)은 대상 밖.
- → **실제 발송은 대표 PC(윈도우)에서 도는 로컬 컴포넌트**가 해야 한다. 클라우드 엔진은 **"안전한 발송 계획·명단 판정·속도·승인·기록"**만 맡고, **발송 행위는 로컬**이 한다. (역할 분담 = 4절)
- ❓ **결정 필요:** 로컬 컴포넌트를 (a) 별도 작은 프로그램으로 새로 만들지, (b) 검증된 기존 도구(나인톡 등)를 쓰되 우리 엔진이 안전한 명단·문구만 공급할지.

### 0-B. 정직한 트레이드오프 — 계정 위험 (1번 원칙과 직결)
| | 나인톡 모델(개인 카톡) | 공식 API(친구톡/알림톡, Solapi) |
|---|---|---|
| 계정 정지 위험 | **있음**(개인계정 자동화=카카오 정책 위반 소지, "100% 보장 없음"=코치 확인) | **없음**(공식 비즈메시지) |
| 도달 | 대화이력 있는 고객 OK | 채널 친구(친구톡)/템플릿(알림톡) |
| 비용·승인 | 무료·즉시 | 건당 비용·채널연동/템플릿승인 |
- ★1번 원칙("계정 안 잃는다")에 가장 부합하는 건 **공식 API**(위험 0). 나인톡 모델은 **본질적으로 위험을 안고** 가는 길(안전장치로 줄일 뿐 0은 아님).
- 대표가 객관 검증 후 나인톡 모델을 택하셨으니 그대로 설계하되, **이 위험을 문서에 박아두고** 안전장치를 이중삼중으로 건다. (원하시면 "정보성=알림톡(위험0) + 관계형=나인톡" 하이브리드도 가능 — 6절 질문.)

## 1. 나인톡 모델 — 무엇을 하나
대표의 **기존 고객(대화 이력 있는 사람)**에게, 대표 본인 카톡으로 **1:1 개인톡을 한 명씩 천천히** 보낸다. 대표는 **1클릭으로 시작**, 나머지는 사람처럼 순차 진행. **새 채팅방은 절대 안 만든다.**

---

## 2. ★ 반드시 박는 안전 설계 (1번 원칙 구현)

### ① 대화 이력 있는 고객만 — 미지 명단 시스템 차단 (새 채팅방 0)
- **기존 대화방이 있는 상대에게만** 발송. **처음 보는 번호/명단 = 시스템이 거부**(새 채팅방 생성 0). → 대량 신규방 = 이용자보호조치 트리거를 **원천 차단.**
- 판정: 로컬 컴포넌트가 "이 상대와 기존 대화방이 있는가"를 확인 후에만 전송. 없으면 **건너뜀+사유 기록**(절대 새로 안 엶).

### ② 사람처럼 천천히 — 시간 단위 소량·상한
- **건당 간격 30~40초**(랜덤 지터)·**시간당 상한**·**하루 상한**. 한 번에 몰아치기 금지.
- 야간(21~08) 발송 금지. 사람 활동 시간대만.

### ③ 본인 승인·1클릭·중단 가능
- 발송 전 **미리보기**(누구에게 N명·문구·제외자) → **대표 1클릭 시작**. 자동 시작 영영 0.
- **언제든 즉시 정지(킬 스위치).** 이상 신호(전송 실패 급증·카카오 경고) 감지 시 **자동 일시정지+대표 호출**(자동 우회 0 — AI손 L2 실패카드와 동일 철학).

### ④ 감사·동의·개인정보
- 모든 시도 **감사 로그**(상대·시각·성공/건너뜀/실패·사유). 수신거부(optout) 명단 자동 제외. 번호 마스킹 보관.

## 3. 역할 분담 (클라우드 엔진 / 로컬 컴포넌트)

| 일 | 누가 | 내용 |
|---|---|---|
| 명단·문구 준비, 안전 판정(동의·optout), 순서·속도 계획, 미리보기, 승인 접수, 감사로그 | **클라우드 엔진(jenya)** | 발송 0 — 계획·기록만 |
| "기존 대화방 있나" 확인, 실제 1:1 전송, 속도 지터, 킬스위치, 결과 회신 | **로컬 컴포넌트(대표 PC)** | 실제 카톡 조작 |
- 엔진은 **여전히 발송 0**(발행 PROTECT·회귀 0 유지). 위험한 행위는 전부 로컬·사람 통제 하에.

## 5. 착수 순서 (코치 승인 후 — 0-A/0-B 결정이 먼저)

- **선결:** 0-A(로컬 컴포넌트 어디서·새로/기존도구) + 0-B(나인톡 단독 vs 알림톡 하이브리드) 확정.
- **1 (엔진 골격):** `POST /bulk/plan`(안전 판정·미리보기·발송0) + 발송계획/감사 시트 + optout 제외. ★클라우드는 여기까지만(발송 0).
- **2 (로컬 컴포넌트 PoC):** 대표 PC에서 **기존 대화방 1명**에게 1건, 40초 간격·킬스위치 실증(★대화이력 없는 상대 거부 먼저 증명).
- **3:** 엔진 계획 ↔ 로컬 실행 연결 + 감사 회신 + 이상신호 자동정지.
- **검증:** ★★ N1~N10 → 전부 O 후에만 실사용. 소수(예: 5명)부터 점진.

## 6. 코치 결정 필요 (구현 전 확정)

1. **0-A 로컬 컴포넌트:** (a) 우리가 작은 로컬 프로그램 신규 제작 vs (b) 검증된 기존 도구(나인톡 등) 사용 + 엔진은 안전 명단·문구·계획만 공급? — ★1번 원칙상 "기존 검증 도구 + 우리 안전계획"이 더 안전할 수 있음.
2. **0-B 위험 수용:** 나인톡 모델 단독 vs **정보성=알림톡(위험0)+관계형=나인톡** 하이브리드? (계정 위험 최소화 관점.)
3. **대상 명단:** "대화 이력 있는 고객" 판정 = 로컬에서만 가능(엔진은 모름). 엔진은 후보 명단만 주고 **로컬이 최종 필터**하는 구조 OK?
4. **속도·상한:** 건당 40초·시간당 N·하루 N 구체값?
5. **scope:** 대표(OWNER) 본인 카톡 먼저 vs 교육생 각자 카톡(교육생은 각자 PC·각자 위험 — 훨씬 신중).
