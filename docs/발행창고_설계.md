# 발행창고 설계

> # ★ 자동화의 끝은 발행이 아니라 "대표님 앞에 놓기"다.
>
> 1차 파이프라인이 멈춘 이유가 정확히 이것이다. 쇼츠 5편을 만들어뒀지만
> 대표님이 **퀄리티 미달로 판정**하셔서 한 편도 못 썼다. 그때 자동이었다면
> 퀄리티 미달이 **46,200명 채널에 그대로 나갔다.**
>
> 만드는 건 기계가 한다. **내보낼지는 사람이 정한다.** 이 문장이 흔들리면
> 아래 설계는 전부 무의미하다.

*(2026-07-16 확정. 1단계 라이브 = https://jenya.onrender.com/warehouse)*

## 0. ⚠️ 구조 정정 — 지금 라이브 화면은 틀렸다 (2026-07-16, 아직 안 고침)

**v5 발행이 최우선이라 기록만 해둔다. 발행 끝나고 고친다.**

### 틀린 것
목업·현재 라이브 화면: **"유형 = 진단페이지 / 랜딩페이지" 중 선택** (4단)
→ **랜딩은 선택 대상이 아니다. 랜딩은 항상 최종 목적지다.**

### 맞는 구조

```
콘텐츠(쇼츠) → 진단(그물) → 랜딩(목적지) → 고객
```

- **랜딩 = 서비스 = 4개** (재무상담 / 부트캠프 / 일반인강의 / 전문가강의)
- **진단 = 그 랜딩으로 들어오게 하는 유입구**

| 서비스 | 랜딩 : 진단 | 진단 목록 |
|---|---|---|
| **재무상담** | **1 : 6** (1대多) | 연금분석 ⭐지금 / 금융집짓기 / 수입지출 / 자산부채 / 금융오행 / DESIRE |
| 부트캠프 | 1 : 1 | 표준진단 → 콘텐츠 = 1인사업자를 위한 AI에이전트 |
| 일반인강의 | 1 : 1 | 표준진단 → 콘텐츠 = 10억목돈마련절대법칙 |
| 전문가강의 | 1 : 1 | 표준진단 → 콘텐츠 = 금융집짓기 |

### 선택은 3단이다 (4단 아님)

```
① 서비스명   재무상담 / 부트캠프 / 일반인강의 / 전문가강의
             → 랜딩 URL이 자동 결정 (선택 아님. 표시만)
② 진단       ①에 따라 목록이 바뀜
③ 한줄카피   [+업로드]
```

- **도착지 = 진단 URL** — 쇼츠가 보내는 곳. 꼬리표(`?from=`)가 붙는 곳.
- **최종 목적지 = 랜딩 URL** — 진단이 끝나고 가는 곳. **표시만 한다.**

### 시트 「캠페인」 탭 — 새 컬럼과 씨앗 9행

```
서비스 | 진단명 | 콘텐츠주제 | 코드 | 진단URL(도착지) | 랜딩URL(최종목적지) | 상태
```

**★`콘텐츠주제` 컬럼이 필요한 이유:** 표준진단 3개가 **이름은 같은데 콘텐츠가 다르다.**
대표님이 헷갈리시면 안 된다.

| # | 서비스 | 진단명 | 콘텐츠주제 | 코드 | 진단URL(도착지) | 랜딩URL(최종목적지) | 상태 |
|---|---|---|---|---|---|---|---|
| 1 | 재무상담 | 연금분석 | | `pen` | `…/연금진단_시뮬레이터_v4.html` ✅ | `…/consult.html` ✅ | **활성** ⭐진행중 |
| 2 | 부트캠프 | 표준진단 | 1인사업자를 위한 AI에이전트 | `agt` | (확인필요) | (확인필요) | 대기 |
| 3 | 일반인강의 | 표준진단 | 10억목돈마련절대법칙 | `bil` | (확인필요) | `ohwant-lecture.netlify.app` ✅ | 대기 |
| 4 | 전문가강의 | 표준진단 | 금융집짓기 | `hou` | (확인필요) | (확인필요) | 대기 |
| 5 | 재무상담 | DESIRE | | `des` | `…/desire.html` ✅ | `…/consult.html` ✅ | **활성** |
| 6 | 재무상담 | 금융집짓기 | | `fhb` | (확인필요) | `…/consult.html` ✅ | 대기 |
| 7 | 재무상담 | 금융오행 | | `ohg` | (확인필요) | `…/consult.html` ✅ | 대기 |
| 8 | 재무상담 | 수입지출 | | `inc` | (확인필요) | `…/consult.html` ✅ | 대기 |
| 9 | 재무상담 | 자산부채 | | `ast` | (확인필요) | `…/consult.html` ✅ | 대기 |

**확실한 것만 넣는다. 나머지 (확인필요)는 대표님이 시트에서 직접 채우신다.**

### ★ 활성 판정 — 진단URL만 본다

- **진단URL이 없으면 활성 불가.** 쇼츠가 보낼 곳이 없으니 발행 자체가 무의미하다.
- **랜딩URL은 없어도 발행된다.** 쇼츠는 **진단**으로 보내니까. 랜딩은 **표시만**이다.
- "주소가 있어야 활성" 원칙은 그대로 — 다만 **그 주소는 진단URL**이다.
  (상태 칸에 '활성'이라고 적혀 있어도 진단URL이 비면 `live=false`)

### 고칠 곳 4군데

1. 시트 컬럼 (`유형`·`내용` → `진단명`·`콘텐츠주제`, `랜딩URL` 신설, 코드 재발급)
2. `server/warehouse.js` — `readCampaigns()`의 컬럼 매핑과 `live` 규칙
3. `발행창고.html` — 선택 4단 → **3단**, 랜딩은 표시만
4. 이 문서

**기존 코드(`con`·`lec`·`boo`·`pro`)는 없어진다.** `agt`·`fhb`·`ohg`·`inc`·`ast`가 새로 생긴다.

## 1. 전체 흐름

```
① 한줄카피   대표님이 올림 (txt, 캠페인당 60개)
     │
     ▼
② 원고       ★서버(jenya)가 LLM으로 생성. PC 안 거침.
     │        → 대표님이 주무셔도 원고 60개가 쌓인다.
     ▼
③ 콘텐츠     ★대표님 PC 러너(엄마1)가 GPU로 제작 → 파이어베이스 업로드
     │
     ▼
🚦 검수       ★대표님이 눈으로 보고 승인/반려.  ← 자동화의 끝
     │
     ▼
④ 발행       서버가 예약표에 넣음 → 기존 발행 스케줄러가 발행
```

**캠페인 = 서비스 × 유형 × 내용.** 캠페인 하나 = 카피 60개.

## 2. 역할 — 무엇이 어디서 도는가

| 단계 | 어디서 | 왜 |
|---|---|---|
| ② 원고 | **서버** | LLM이라 GPU가 필요 없다. PC로 보내면 PC가 켜져 있어야만 원고가 나온다 |
| ↳ 프롬프트 | **코치가 씀** | `원고규칙.md`의 8개 규칙이 그대로 프롬프트가 된다. 우리가 안 만든다 |
| ③ 콘텐츠 | **PC 러너** | 음성 클론·영상 합성은 GPU가 필요하다. 서버엔 GPU가 없다 |
| 🚦 검수 | **대표님** | 눈으로 하는 일 |
| ④ 발행 | **서버** | 이미 다 만들어져 있다(유튜브·인스타 직접발행 + 60초 스케줄러) |

**서버는 PC 안을 못 본다. PC는 서버를 부를 수 있다.** → PC가 서버를 폴링한다.
(서버가 PC를 부르는 길은 없다 — PC에 공개 주소가 없다.)

## 3. 상태 기계 — 진실은 시트

`한줄카피` 탭의 `상태` 칸 하나로 전부 표현한다.

```
대기 ──(서버 LLM)──▶ 원고완료 ──(러너 claim)──▶ 제작중
                          ▲                        │ (완성 + 파이어베이스 업로드)
                          │                        ▼
                     (반려·시드 재추첨)         검수대기
                          │                        │
                          └────────────────────────┤ (대표님)
                                                   ▼
   발행완료 ◀──(기존 스케줄러)── 예약됨 ◀──(승인)──┘

  원고수정필요 ◀── (반려 사유 = 원고 문제 / 반려 3회 초과)
```

**★대표님이 시트에서 손으로 상태를 바꿔도 된다.** `검수대기`를 `승인`으로
바꾸시면 그게 승인이다. **화면 버튼은 편의고, 진실은 시트다.**
화면이 죽어도 대표님이 진행하실 수 있어야 한다.

## 4. 러너 ↔ 서버 계약

**러너는 시트를 직접 안 읽는다. 서버 창구를 부른다.**
그래야 ①러너에 SA 열쇠가 안 들어가고 ②서버가 중복 제작을 막고
③시트 컬럼이 바뀌어도 러너를 안 고친다.

```
POST /warehouse/claim   → 서버가 '원고완료' 1건을 '제작중'으로 찍고 넘김
                          (러너ID·시각 기록)
POST /warehouse/done    → 러너가 미디어URL 보고 → '검수대기'
POST /warehouse/fail    → 제작 실패 보고 → '원고완료'로 복귀 + 사유 기록
```

### ★ 중복 제작 방지 (오늘 검증한 패턴 재사용)

"대기 중 첫 번째를 가져간다"를 두 번 하면 **같은 걸 두 번 만든다.**
러너를 재시작하거나 이전 작업이 안 끝났는데 또 돌면 그렇게 된다.
GPU 시간이 두 배로 날아가고 파이어베이스에 쓰레기가 쌓인다.
→ **claim이 상태를 '제작중'으로 찍는다.**
→ **'제작중'이 30분 넘으면 '대기'로 자동 복귀**(러너가 죽었을 때).
   발행 스케줄러의 `STUCK_MIN`과 같은 장치다.

## 5. 파이어베이스 업로드 — 러너에 열쇠 0

브라우저가 쓰는 그 길을 러너가 그대로 쓴다:

```
① POST /content/upload-url  → 서버가 서명 URL 발급(v4, 15분)
② PUT  파일 → 버킷 직행       ← 서버가 파일 바이트를 안 거친다
③ POST /content/uploaded    → 공개 다운로드 토큰 → 링크 완성
```

**서버가 열쇠를 쥐고 서명만 해준다.** 러너엔 SA 키가 없다.
HTTP 세 번이라 파이썬이든 노드든 상관없다.

## 6. 열쇠 — `WAREHOUSE_RUNNER_KEY` (러너 전용, 새로 판다)

- **발행용 `CRON_KEY`를 재사용하지 않는다.** 하나 새면 둘 다 뚫린다.
- **"열쇠 0" 원칙과 충돌하지 않는다.** 우리가 막은 건
  *"SA 키를 GitHub 같은 남의 서비스에 꺼내는 것"*이었다.
  러너는 **대표님 PC**에 있다. 대표님 것이다. 남이 아니다.
- Render 환경변수 + 러너 설정 파일에 각각 둔다.

**⚠️ 숙제:** 지금 `/content/upload-url`·`/content/uploaded`는 **아무 보호가 없다.**
주소만 알면 누구나 우리 버킷에 올릴 수 있다. 러너를 붙일 때 같이 막아야 한다.

## 7. 검수 화면 (발행창고 안)

시트로는 영상을 못 본다. **검수는 눈으로 하는 일이다.**

- 쇼츠 영상 미리보기 — 파이어베이스 URL 그대로 `<video>`
- 원고 전문 — 숫자가 맞는지 눈으로
- **[승인] [반려]** 버튼
- 반려 시 **사유 드롭다운**(아래) + 한 줄 메모(선택)
- 폰에서 되어야 한다. 대표님은 폰으로 보신다.

## 8. 반려 루프

**같은 원고로 다시 만들면 또 같은 게 나온다**(시드만 다르다).
그래서 **반려 사유가 러너에게 무엇을 하라고 말해줘야 한다.**

**★사유는 자유 텍스트가 아니라 드롭다운이다.** 자유 텍스트면 러너가 해석을 못 한다.
("사람이 규칙을 지키게 하는 설계는 실패한다" — 대표님, 2026-07-16)

| 반려 사유 | 러너가 하는 일 |
|---|---|
| 목소리 이상 | 시드 재추첨 → 자동 재제작 |
| 영상 이상 | 시드 재추첨 → 자동 재제작 |
| 자막 오류 | 시드 재추첨 → 자동 재제작 |
| **원고 문제** | **아무것도 안 한다.** 상태 = `원고수정필요` → 사람(코치)이 고친다 |
| 기타 | 상태 = `원고수정필요` (사람이 판단) |

**★반려 3회 초과 → `원고수정필요` + 대표님께 알림.**
> "이 카피는 원고를 손봐야 합니다"

시드를 아무리 다시 뽑아도 안 되는 건 **원고가 문제**다. 그건 자동화 밖이다.
무한 재제작으로 GPU를 태우지 않는다.

## 9. 러너 실행 — 대표님이 켤 때만

배치는 밤새 도는 물건이지 24시간 상시일 필요가 없다.
GPU 13~25시간짜리를 상시로 두면 PC가 계속 뜨겁다.

**그림:** 대표님이 저녁에 켜고 → 주무시고 → 아침에 결과를 검수하신다.

**PC가 꺼져 있으면 ③이 멈추는 게 정상이다.** 고장이 아니다.
그때 **재고 감시기**가 알려준다(설계 완료·코드는 발행 성공 후).

**★러너는 콘솔에 "지금 몇 편 남았고 언제 끝나는지"를 계속 보여준다.**
대표님이 자기 전에 한 번 보시고 안심하실 수 있게.

## 9-1. `/warehouse/done` 규격 — 러너(엄마1) ↔ 서버 계약

자동 검수단 4명(원고·용어·음성·화면)이 편마다 판정한다. 러너가 그 결과를 이 창구로 보낸다.
**540편이면 사람이 못 본다.** 대표님은 요약만 보고 3초 안에 판단하신다.

### ★ 코치 규격안에서 한 군데를 바꿉니다 — 러너가 `최종상태`를 정하면 안 된다

원안: `{ 카피번호, 검수관별 판정, 측정값, 재시도 횟수, 최종상태 }`

**`최종상태`를 러너가 보내면, 러너의 버그 하나가 `승인`을 만들 수 있다.**
그러면 검수를 안 거친 영상이 예약표로 넘어가고 **46,200명 채널로 나간다.**

> **검수 통과 = `검수대기`이지 `승인`이 아니다. 승인은 대표님만 한다.**
> 러너는 **판정 결과만** 보고한다. **상태는 서버가 정한다.**

이게 이 문서 맨 위 원칙("자동화의 끝은 대표님 앞에 놓기")을 코드로 지키는 지점이다.

**재시도 횟수도 서버가 센다.** 러너가 죽었다 살아나면 자기 카운트를 잃는다. **시트가 진실이다.**

```json
    { "name": "용어", "pass": false, "detail": "'해지연금'은 없는 말입니다 (퇴직연금 오인)" },
    { "name": "음성", "pass": true,  "detail": "무음 0.4s · 속도 1.05x" },
    { "name": "화면", "pass": true,  "detail": "자막 잘림 없음" }
  ],
  "note": "시드 재추첨 후 재제작"
}
```

| 칸 | 왜 필요한가 |
|---|---|
| `campaign` | **번호만으론 겹친다.** 캠페인이 9개다 — `pen_001`과 `agt_001`은 다른 물건 |
| `runId` | **멱등키.** 러너가 재시도하며 같은 보고서를 두 번 보내도 한 번만 반영된다 |
| `mediaUrl` | 없으면 제작 실패로 본다. 검수는 통과했는데 파일이 없으면 말이 안 된다 |
| `attempt` | 러너가 아는 시도 번호. **참고용** — 진실은 서버 카운트 |
| `checks` | **★배열이다.** 검수관이 5명이 돼도 서버를 안 고친다. `name`은 러너가 정한다 |
| `note` | 선택. 사람이 읽을 한 줄 |

**인증 = Basic Auth**(`lockWrite`). 이미 있는 그 자물쇠를 쓴다. 러너 전용 열쇠를 따로 안 판다.

### 서버가 정하는 것 (러너는 못 정한다)

```
checks 전부 pass  AND  mediaUrl 있음   →  검수대기      (대표님 앞에 놓기)
하나라도 fail  AND  누적 3회 미만       →  원고완료      (러너가 시드 재추첨해 다시)
하나라도 fail  AND  누적 3회 이상       →  원고수정필요  + 알림 (사람이 원고를 고쳐야)
```

### 응답

```json
{ "ok": true, "status": "검수대기", "fails": 0, "retries": 1, "stop": true }
```
- `stop: true` → **러너는 이 카피를 그만 시도한다.** 통과했거나(검수대기),
  3회를 넘겼거나(원고수정필요). 무한 재제작으로 GPU를 태우지 않는다.
- `stop: false` → 다시 시도해도 된다.

### 실패 보고 — `POST /warehouse/fail`

검수 이전에 **제작 자체가 터졌을 때**(GPU 오류·파일 깨짐 등).
```json
{ "campaign": "…", "no": "001", "runId": "…", "why": "ffmpeg 종료코드 1" }
```
→ 서버: `원고완료`로 되돌리고 사유 기록. 이것도 3회 넘으면 `원고수정필요`.

### 시트 컬럼 추가 (`한줄카피` 탭)

```
… | 미디어URL | 검수 | 재시도 | 검수상세
```
- **`검수`** = 사람이 읽는 한 줄. `✅ 4/4 합격` / `⚠️ 용어 불합격`
- **`검수상세`** = `checks` 원본 json. 화면이 펴서 보여준다. 대표님은 안 읽으셔도 된다

## 9-2. 검수 화면 — 대표님이 3초 안에 판단하시게

```
┌────────────────────────────────────────────────┐
│ [영상]   001  국민연금 68만원, 이걸로 노후…      │
│                                                │
│   ✅ 4명 전부 합격 · 재시도 1회                 │
│   ▸ 상세                        [승인] [반려]  │
└────────────────────────────────────────────────┘

┌────────────────────────────────────────────────┐
│ [영상]   007  퇴직연금 그냥 두면…               │
│                                                │
│   ⚠️ 용어 검수관 불합격 3회 → 원고 수정 필요     │
│   ▸ 상세                      [승인] [반려]    │
│                                ↑ 회색·못 누름  │
└────────────────────────────────────────────────┘
```

- **★불합격 3회짜리는 [승인]을 회색으로.** 실수로 못 누르시게.
  검수단이 세 번 잡아낸 걸 사람이 실수로 통과시키면 검수단이 있으나 마나다.
- 상세는 **접었다 펴기.** 펴면 검수관별 판정·측정값.
- **반려 사유는 드롭다운**(§8) — 자유 텍스트는 러너가 못 읽는다.

## 10. 러너의 입력 = 원고 json (밖에서 온다)

**러너는 원고가 어디서 왔는지 신경 쓰지 않는다.** 서버가 LLM으로 만들든, 코치가
손으로 넣든, 원고 json이 창구로 오면 러너는 그걸로 만들 뿐이다.

→ 러너 설계는 **"원고 json이 밖에서 온다"**고만 가정한다.
→ 그래서 ② 원고 생성을 나중에 바꿔도 러너를 안 고친다.

프롬프트는 **코치가 쓴다** — 엄마1이 `원고규칙.md`에 박아둔 8개 규칙이 그대로 프롬프트가 된다.

## 아직 안 정한 것

- ~~폰에서 파이어베이스 mp4 재생~~ → **발행창고 화면에 실제 mp4를 걸어 대표님이 폰으로 실측.**
  추측하지 말고 눌러보는 게 빠르다.
- **반려 알림 = 디스코드 웹훅**(대표님이 만드시면). 재고 감시기와 같은 경로.
  **그때까지는 화면 배지로** 표시한다(알림 없이도 대표님이 화면에서 보실 수 있게).
- ② 원고의 채널별 규격(쇼츠/릴스/캐러셀/블로그) — `원고규칙.md` 나오면 확정

## 이 설계가 지키는 것

1. **자동화의 끝은 대표님 앞에 놓기** (맨 위)
2. 진실은 시트, 화면 버튼은 편의
3. 열쇠는 서버 안에만 (러너는 대표님 PC = 예외)
4. 조용히 잘못되느니 시끄럽게 멈춘다
5. 없는 걸 있는 척하지 않는다
