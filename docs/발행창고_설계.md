# 발행창고 설계

> # ★ 자동화의 끝은 발행이 아니라 "대표님 앞에 놓기"다.
>
> 1차 파이프라인이 멈춘 이유가 정확히 이것이다. 쇼츠 5편을 만들어뒀지만
> 대표님이 **퀄리티 미달로 판정**하셔서 한 편도 못 썼다. 그때 자동이었다면
> 퀄리티 미달이 **46,200명 채널에 그대로 나갔다.**
>
> 만드는 건 기계가 한다. **내보낼지는 사람이 정한다.** 이 문장이 흔들리면
> 아래 설계는 전부 무의미하다.

*(2026-07-16 확정. 1단계 라이브 = https://jenya.onrender.com/warehouse)*

---

## 1. 전체 흐름

```
① 한줄카피   대표님이 올림 (txt, 캠페인당 60개)
     │
     ▼
② 원고       ★서버(jenya)가 LLM으로 생성. PC 안 거침.
     │        → 대표님이 주무셔도 원고 60개가 쌓인다.
     ▼
③ 콘텐츠     ★대표님 PC 러너(엄마1)가 GPU로 제작 → 파이어베이스 업로드
     │
     ▼
🚦 검수       ★대표님이 눈으로 보고 승인/반려.  ← 자동화의 끝
     │
     ▼
④ 발행       서버가 예약표에 넣음 → 기존 발행 스케줄러가 발행
```

**캠페인 = 서비스 × 유형 × 내용.** 캠페인 하나 = 카피 60개.

## 2. 역할 — 무엇이 어디서 도는가

| 단계 | 어디서 | 왜 |
|---|---|---|
| ② 원고 | **서버** | LLM이라 GPU가 필요 없다. PC로 보내면 PC가 켜져 있어야만 원고가 나온다 |
| ↳ 프롬프트 | **코치가 씀** | `원고규칙.md`의 8개 규칙이 그대로 프롬프트가 된다. 우리가 안 만든다 |
| ③ 콘텐츠 | **PC 러너** | 음성 클론·영상 합성은 GPU가 필요하다. 서버엔 GPU가 없다 |
| 🚦 검수 | **대표님** | 눈으로 하는 일 |
| ④ 발행 | **서버** | 이미 다 만들어져 있다(유튜브·인스타 직접발행 + 60초 스케줄러) |

**서버는 PC 안을 못 본다. PC는 서버를 부를 수 있다.** → PC가 서버를 폴링한다.
(서버가 PC를 부르는 길은 없다 — PC에 공개 주소가 없다.)

## 3. 상태 기계 — 진실은 시트

`한줄카피` 탭의 `상태` 칸 하나로 전부 표현한다.

```
대기 ──(서버 LLM)──▶ 원고완료 ──(러너 claim)──▶ 제작중
                          ▲                        │ (완성 + 파이어베이스 업로드)
                          │                        ▼
                     (반려·시드 재추첨)         검수대기
                          │                        │
                          └────────────────────────┤ (대표님)
                                                   ▼
   발행완료 ◀──(기존 스케줄러)── 예약됨 ◀──(승인)──┘

  원고수정필요 ◀── (반려 사유 = 원고 문제 / 반려 3회 초과)
```

**★대표님이 시트에서 손으로 상태를 바꿔도 된다.** `검수대기`를 `승인`으로
바꾸시면 그게 승인이다. **화면 버튼은 편의고, 진실은 시트다.**
화면이 죽어도 대표님이 진행하실 수 있어야 한다.

## 4. 러너 ↔ 서버 계약

**러너는 시트를 직접 안 읽는다. 서버 창구를 부른다.**
그래야 ①러너에 SA 열쇠가 안 들어가고 ②서버가 중복 제작을 막고
③시트 컬럼이 바뀌어도 러너를 안 고친다.

```
POST /warehouse/claim   → 서버가 '원고완료' 1건을 '제작중'으로 찍고 넘김
                          (러너ID·시각 기록)
POST /warehouse/done    → 러너가 미디어URL 보고 → '검수대기'
POST /warehouse/fail    → 제작 실패 보고 → '원고완료'로 복귀 + 사유 기록
```

### ★ 중복 제작 방지 (오늘 검증한 패턴 재사용)

"대기 중 첫 번째를 가져간다"를 두 번 하면 **같은 걸 두 번 만든다.**
러너를 재시작하거나 이전 작업이 안 끝났는데 또 돌면 그렇게 된다.
GPU 시간이 두 배로 날아가고 파이어베이스에 쓰레기가 쌓인다.
→ **claim이 상태를 '제작중'으로 찍는다.**
→ **'제작중'이 30분 넘으면 '대기'로 자동 복귀**(러너가 죽었을 때).
   발행 스케줄러의 `STUCK_MIN`과 같은 장치다.

## 5. 파이어베이스 업로드 — 러너에 열쇠 0

브라우저가 쓰는 그 길을 러너가 그대로 쓴다:

```
① POST /content/upload-url  → 서버가 서명 URL 발급(v4, 15분)
② PUT  파일 → 버킷 직행       ← 서버가 파일 바이트를 안 거친다
③ POST /content/uploaded    → 공개 다운로드 토큰 → 링크 완성
```

**서버가 열쇠를 쥐고 서명만 해준다.** 러너엔 SA 키가 없다.
HTTP 세 번이라 파이썬이든 노드든 상관없다.

## 6. 열쇠 — `WAREHOUSE_RUNNER_KEY` (러너 전용, 새로 판다)

- **발행용 `CRON_KEY`를 재사용하지 않는다.** 하나 새면 둘 다 뚫린다.
- **"열쇠 0" 원칙과 충돌하지 않는다.** 우리가 막은 건
  *"SA 키를 GitHub 같은 남의 서비스에 꺼내는 것"*이었다.
  러너는 **대표님 PC**에 있다. 대표님 것이다. 남이 아니다.
- Render 환경변수 + 러너 설정 파일에 각각 둔다.

**⚠️ 숙제:** 지금 `/content/upload-url`·`/content/uploaded`는 **아무 보호가 없다.**
주소만 알면 누구나 우리 버킷에 올릴 수 있다. 러너를 붙일 때 같이 막아야 한다.

## 7. 검수 화면 (발행창고 안)

시트로는 영상을 못 본다. **검수는 눈으로 하는 일이다.**

- 쇼츠 영상 미리보기 — 파이어베이스 URL 그대로 `<video>`
- 원고 전문 — 숫자가 맞는지 눈으로
- **[승인] [반려]** 버튼
- 반려 시 **사유 드롭다운**(아래) + 한 줄 메모(선택)
- 폰에서 되어야 한다. 대표님은 폰으로 보신다.

## 8. 반려 루프

**같은 원고로 다시 만들면 또 같은 게 나온다**(시드만 다르다).
그래서 **반려 사유가 러너에게 무엇을 하라고 말해줘야 한다.**

**★사유는 자유 텍스트가 아니라 드롭다운이다.** 자유 텍스트면 러너가 해석을 못 한다.
("사람이 규칙을 지키게 하는 설계는 실패한다" — 대표님, 2026-07-16)

| 반려 사유 | 러너가 하는 일 |
|---|---|
| 목소리 이상 | 시드 재추첨 → 자동 재제작 |
| 영상 이상 | 시드 재추첨 → 자동 재제작 |
| 자막 오류 | 시드 재추첨 → 자동 재제작 |
| **원고 문제** | **아무것도 안 한다.** 상태 = `원고수정필요` → 사람(코치)이 고친다 |
| 기타 | 상태 = `원고수정필요` (사람이 판단) |

**★반려 3회 초과 → `원고수정필요` + 대표님께 알림.**
> "이 카피는 원고를 손봐야 합니다"

시드를 아무리 다시 뽑아도 안 되는 건 **원고가 문제**다. 그건 자동화 밖이다.
무한 재제작으로 GPU를 태우지 않는다.

## 9. 러너 실행 — 대표님이 켤 때만

배치는 밤새 도는 물건이지 24시간 상시일 필요가 없다.
GPU 13~25시간짜리를 상시로 두면 PC가 계속 뜨겁다.

**그림:** 대표님이 저녁에 켜고 → 주무시고 → 아침에 결과를 검수하신다.

**PC가 꺼져 있으면 ③이 멈추는 게 정상이다.** 고장이 아니다.
그때 **재고 감시기**가 알려준다(설계 완료·코드는 발행 성공 후).

**★러너는 콘솔에 "지금 몇 편 남았고 언제 끝나는지"를 계속 보여준다.**
대표님이 자기 전에 한 번 보시고 안심하실 수 있게.

---

## 10. 러너의 입력 = 원고 json (밖에서 온다)

**러너는 원고가 어디서 왔는지 신경 쓰지 않는다.** 서버가 LLM으로 만들든, 코치가
손으로 넣든, 원고 json이 창구로 오면 러너는 그걸로 만들 뿐이다.

→ 러너 설계는 **"원고 json이 밖에서 온다"**고만 가정한다.
→ 그래서 ② 원고 생성을 나중에 바꿔도 러너를 안 고친다.

프롬프트는 **코치가 쓴다** — 엄마1이 `원고규칙.md`에 박아둔 8개 규칙이 그대로 프롬프트가 된다.

## 아직 안 정한 것

- ~~폰에서 파이어베이스 mp4 재생~~ → **발행창고 화면에 실제 mp4를 걸어 대표님이 폰으로 실측.**
  추측하지 말고 눌러보는 게 빠르다.
- **반려 알림 = 디스코드 웹훅**(대표님이 만드시면). 재고 감시기와 같은 경로.
  **그때까지는 화면 배지로** 표시한다(알림 없이도 대표님이 화면에서 보실 수 있게).
- ② 원고의 채널별 규격(쇼츠/릴스/캐러셀/블로그) — `원고규칙.md` 나오면 확정

## 이 설계가 지키는 것

1. **자동화의 끝은 대표님 앞에 놓기** (맨 위)
2. 진실은 시트, 화면 버튼은 편의
3. 열쇠는 서버 안에만 (러너는 대표님 PC = 예외)
4. 조용히 잘못되느니 시끄럽게 멈춘다
5. 없는 걸 있는 척하지 않는다
