# 단체카톡 (B) 로컬 하드가드 체커 — L3 설계 검토안

> 작성: 2026-06-21 (엔진=엄마 / 코치 검토·승인 대기). **구현 전 설계 문서 — 검토안.**
> ★★★ 1번 원칙: **회원 한 명의 카톡 계정도 잃지 않는다.** (B) = 시판 도구에 **없는** "대화방 없으면 자동 제외"를 **우리가** 보장.
> 전제: 엔진 골격(plan·approve·result·audit) 완성·발송 0. (B)는 **대표 PC에서 도는 로컬 컴포넌트**(클라우드와 분리).

---

## 0. (B)가 하는 일 — 한 줄
대표 PC에서, **엔진이 내보낸 승인 발송계획**을 받아 → **각 상대에게 기존 카톡 대화방이 실제로 있는지 확인** → **없으면 자동 제외(새 방 0)** → 있는 사람만 **40초 간격·킬스위치**로 발송 → 결과를 엔진(`/bulk/result`)에 회신.

```
엔진 export(승인 명단·문구·속도) → [로컬 (B)] 대화방 존재 확인(하드가드) → 있는 사람만 발송 → /bulk/result
```

---

## 1. ★먼저 짚을 현실 — "대화방 존재 확인"은 왜 어렵나 (정직)
카카오톡은 **공식 API가 없다.** "이 번호와 기존 대화방이 있나"를 프로그램이 아는 방법이 마땅치 않다:
- **(가) 카톡 PC 로컬 DB 읽기** — 암호화·비공식·업데이트마다 깨짐. 비권장(불안정·약관 위험).
- **(나) GUI로 검색해 확인** — 카톡 PC에서 상대 검색 → "대화방이 이미 있나" 화면으로 판별. 가능하나 느리고 화면 변화에 취약.
- **(다) 사전 추출 allowlist(권장)** — 대표가 **본인 카톡의 "기존 대화 상대" 목록을 1회 추출**(이름/번호) → (B)는 발송 대상이 **그 목록에 있을 때만** 전송, 없으면 제외. "새 방 생성"을 코드 흐름에서 원천 차단.
→ **권장 = (다) 사전 추출 allowlist + (나) 발송 직전 화면 재확인(가능하면).** (가)는 안 씀.

---

## 2. 3중 방어로 정리 (엔진 + 로컬 이중을 더 쪼갬)
| 층 | 누가 | 내용 |
|---|---|---|
| 1차 | **엔진** | 결제·CRM 고객만(known 힌트)·optout·상한·속도정책 (이미 완성) |
| 2차 | **로컬 allowlist** | 대표 카톡 "기존 대화 상대" 추출 목록에 **있는 사람만**(없으면 제외) ← (B) 핵심 |
| 3차 | **로컬 실행 가드** | 전송 직전 대화방 확인 가능하면 재확인 / **새 방 생성 동작 자체를 안 함**·40초·킬스위치·이상정지 |
→ 1·2차에서 "처음 보는 사람"은 두 번 걸러지고, 3차에서 "새 방 안 만듦"이 마지막 잠금.

---

## 3. (B) 로컬 컴포넌트 — 책임·안전
- **입력:** 엔진 `/bulk/approve` export(planId·문구·속도·recipients 실번호) + 대표 카톡 기존대화 allowlist.
- **하드가드:** recipient ∉ allowlist → **건너뜀(skipped:'no-chat')**, 절대 새 방 안 엶.
- **속도:** 40초/건+지터·시간당·하루 상한·야간(21~08) 금지.
- **사람 통제:** 대표 1클릭 시작·**킬스위치 즉시중단**·실패/경고 급증 시 자동 일시정지(우회 0).
- **회신:** 건별 결과(sent/skipped/failed+사유) → 엔진 `/bulk/result`(이상신호 집계·감사로그).
- **개인정보:** 번호 로컬에서만 취급, 로그 최소·마스킹.

---

## 4. 발송 실행 방식 (1-B 준수 — 우리가 발송기 새로 안 만듦)
- 실제 "카톡 입력·전송"은 **검증된 기존 도구**가 담당(1-B). (B)는 그 앞단의 **하드가드 필터·속도·킬스위치·회신**만.
- 즉 (B) = "안전 게이트 + 기존 도구 호출/연동". 기존 도구가 allowlist 입력·속도조절을 받아주면 그걸 쓰고, 안 받아주면 (B)가 **대상 목록을 allowlist로 미리 잘라** 도구에 넘긴다(처음 보는 사람은 도구에 아예 안 들어감).
- ★ 기존 도구 선정 기준(재확인): allowlist 주입 가능 + 속도조절 + (가능하면) 대화방 확인. **최소한 "우리가 자른 allowlist만 발송"이 보장되면 채택 가능.**

## 5. ★★ (B) 안전 체크리스트
| # | 시나리오 | 합격 |
|---|---|---|
| L1 | allowlist에 없는 상대 | **건너뜀·새 방 0**(skipped:no-chat) |
| L2 | allowlist 비었거나 미입력 | 발송 0(전원 보류) |
| L3 | 속도 | 40초+지터·시간당·하루 상한·야간차단 |
| L4 | 시작 | 대표 1클릭만(자동 시작 0) |
| L5 | 킬스위치 | 즉시 중단 |
| L6 | 실패·경고 급증 | 자동 일시정지+대표 호출(우회 0) |
| L7 | 결과 | 건별 /bulk/result 회신·감사로그 |
| L8 | 카톡 로컬 DB | 안 건드림((가) 미사용) |
| L9 | 클라우드 엔진 | 발송 0·발행 회귀 0 유지 |

## 6. 착수(승인 후) — 소수부터 점진
- **0:** 도구 선정(allowlist 주입 보장 확인) + 대표 기존대화 allowlist 1회 추출 방법 확정.
- **1:** (B) 골격 — export 읽기 + allowlist 하드가드(없으면 제외) + dryRun(누가 발송/제외 미리보기, 발송 0).
- **2:** 속도·킬스위치·1클릭 + 기존 도구 연동, **본인에게 1건** 실증.
- **3:** **기존 대화 상대 1명** 실제 1건(40초·킬스위치) → 결과 회신 확인.
- **검증:** L1~L9 → 전부 O 후 소수(5명)부터 점진 확대.

## 7. 코치 결정 필요
1. **대화방 확인 방식:** (다) 사전 allowlist 추출(권장) vs (나) GUI 실시간 확인 추가? — allowlist는 무엇으로 추출(대표 카톡 친구/대화 내보내기)?
2. **기존 도구:** allowlist 주입·속도조절 되는 도구 확정(코치 후보 조사 결과 반영).
3. **(B) 형태:** 별도 작은 로컬 프로그램 vs 기존 도구 설정+우리 allowlist 전처리(엔진이 자른 명단만 도구에 넣기 = 더 단순·안전).
4. **scope:** 대표 PC 먼저(교육생은 각자 PC·각자 위험 → 훨씬 뒤·신중).
