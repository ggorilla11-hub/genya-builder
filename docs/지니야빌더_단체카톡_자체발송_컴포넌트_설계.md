# 단체카톡 자체 발송 컴포넌트 (대표 PC) — 설계 검토안

> 작성: 2026-06-21 (엔진=엄마 / 코치 검토·승인 대기). **구현 전 설계 문서 — 검토안. 설계까지만, 실제 카톡 자동화 코드·실행은 대표 환경에서 극도로 신중히.**
> ★★★ 1번 원칙: 회원 한 명의 카톡 계정도 잃지 않는다. 안전 > 기능.
> 결정: 친구톡·알림톡 **금지**(재론 X)·외부도구(오토톡/나인톡) 배제 → **우리가 카톡 PC앱 GUI 자동화 자체 제작**(개인톡). 발행 PROTECT·회귀 0.

## 0. 법적·회색지대 — 정직 표기 (코치 조사 반영, 변호사 아님)
- 나인톡·오토톡 = 카카오 **공식 인정도, 법적 제재도 없는 회색지대**(도구 차단·소송 사례 없음).
- ★**위험의 본질 = "도구"가 아니라 "발송 패턴"** — **단시간 대량 + 새 채팅방 생성**이 이용자보호조치(계정 제한)를 부른다. **대화이력 있는 친구 + 소량**이면 카카오 제한 사실상 없음(대량도 OK 사례).
- 우리 = **오프라인 10명 소규모·극소량 + 대화이력만** → 나인톡 모델보다 **더 안전**.
- **정직:** 카카오 공식 인정은 **아님(회색)**, **100% 보장 없음.** 단 **대화이력·소량이면 실무 위험 매우 낮음.** → 대표님 **위험 인지·수용** 전제, 가능하면 법무 1회 검토.
- → 설계 원칙: **"회색지대 + 소량 + 대화이력 = 실무 안전"** 을 코드로 강제(대량·새 방을 구조적으로 못 하게).

## 1. 아키텍처 (기존 자산 활용 + 새로 만들 것)
- **재사용(완성):** 엔진 안전두뇌(plan·approve·result·audit, 발송0) + 소팅(segment) + 다층방어 + (B) dryRun 교집합 골격(`local-bulk-checker`).
- **새로 제작:** **대표 PC 로컬 발송기** — safe-list(소팅+안전필터된 명단) → 본인 카톡 PC앱으로 **개별 카톡 발송(텍스트+이미지)**. ★클라우드 엔진은 끝까지 발송 0.

## 2. 기술 방식 — Windows GUI 자동화 라이브러리 (옵션·난이도)
| 방식 | 장점 | 단점 | 난이도 |
|---|---|---|---|
| **AutoHotkey(AHK)** | 카톡 자동화 커뮤니티 검증 多·창/키 제어 쉬움 | 윈도우 전용·별도 설치·복잡 로직 한계 | ★★ |
| **Python pywinauto**(UI Automation) | 요소 단위 접근(검색창·전송 버튼)·견고 | 카톡이 UIA 요소를 잘 노출하는지 불확실 | ★★★ |
| **Python pyautogui+opencv**(이미지 인식) | 화면 버튼 인식·범용 | 해상도·테마·UI변경에 취약 | ★★★ |
| Node nut.js/robotjs | repo가 Node | 윈도우 GUI 자동화는 Python보다 약함 | ★★★★ |
- **권장:** **pywinauto(UIA) 우선 + 이미지 인식 보조**, 또는 AHK. → 어느 쪽이든 **카톡 PC UI가 실제로 어떻게 노출되는지 핸즈온 확인**이 선결(2절 자체가 핸즈온 의존).

## 3. 카톡앱 조작 흐름
1. 카톡 PC앱은 **대표님이 이미 로그인해 켜둔 상태**를 사용 — ★**봇 로그인 X**(자격증명 0접촉).
2. safe-list **1명씩**:
   1. 검색창에 이름/번호 입력 → **기존 대화방이 뜨는지 확인**
   2. ★**기존 대화방 있을 때만 열기.** 없으면(=새 채팅방 생성 화면) → **건너뜀(skip:no-chat), 절대 생성 안 함**
   3. 텍스트 붙여넣기 + 이미지(클립보드) 첨부
   4. 전송 → **40초+지터** 대기 → 다음
3. 매 건 **감사로그** + 결과 → 엔진 `/bulk/result`(이상신호 집계).

## 4. 안전장치 (다층 — 코치 spec, 코드로 강제)
① **대화이력 고객만**(새 채팅방 0) ② **40초+지터·하루 소량 상한·야간(21~08) 차단** ③ **본인 1클릭 시작 + 킬스위치(전역 핫키 즉시정지)** ④ **이상신호**(전송 실패·경고 팝업 감지) **자동 일시정지(우회 0)** ⑤ **감사로그**(건별 성공/건너뜀/실패) ⑥ **봇 로그인 X**(본인 앱) ⑦ **개인정보** 로컬만·마스킹.

## 5. ★ 최난점 정직 고지 — "새 방 0"의 GUI 집행
- **"이 사람과 기존 대화방이 있나"를 GUI에서 판별**하는 것이 **핵심이자 가장 어려운 부분**. 검색 결과에서 "기존 대화방 vs 새로 시작"을 카톡이 명확히 구분해주지 않으면 오판 위험.
- → **dryRun + 핸즈온 검증 필수.** GUI 판별이 불확실하면 **(B) 반자동 allowlist(이미 골격 있음)로 보강** = 대표가 미리 확정한 "대화방 있는 사람"만 대상. **이중 안전.**
- 카톡 UI 변경 시 깨짐(유지보수 부담) — 정직 표기.

## 6. dryRun (발송 0 미리보기) — 가장 먼저·항상 안전
- 카톡 **안 건드리고**, safe-list로 **보낼 순서·40초 간격·예상 소요시간·문구·이미지 참조**를 미리보기 + 각 건 "보낼 예정" 시뮬 로그. **발송 0.**
- `local-bulk-checker`(교집합)에 **순서·간격·시뮬**을 더하면 dryRun 완성(카톡 무접촉).

## 7. 단계 (통제된 파일럿 — 극소량부터)
- **0 dryRun:** 발송 0·카톡 무접촉(순서·간격·시뮬 미리보기).
- **1 핸즈온 검증:** 대표 PC 카톡에서 검색→"기존 대화방 판별"이 GUI로 되는지 **읽기만** 확인.
- **2 본인에게 1건:** 대표 본인 카톡방에 1건 실송(자동화 흐름 검증).
- **3 대화이력 1명:** 실제 기존 대화 1명 1건(40초·킬스위치).
- **4 극소량 파일럿:** 5~10명. 이상 0 확인 후에만 확대. ★**교육생 계정 확대는 안전 입증 전 절대 금지**(각자 계정 위험).
