## PART A — 엔진 라이브 실측 5사실 (지배 규칙)

| # | 사실 | UI/도킹 반영 |
|---|---|---|
| ① | **task 엔티티가 엔진에 없음** | **task 상태 = UI 소유.** 엔진엔 "승인·지시"만 전송. `/task/edit` 엔드포인트 **삭제**. 승인 task 카드는 UI 로컬, 승인 시 `/approve`·`/command`만 호출. |
| ② | **`/chat`(지니야 지시)=대화 응답까지만** | `POST /chat`은 `{reply}` 텍스트만, 부수효과 없음. 실제 행동은 **승인 거쳐 별도**(`/approve`·`/command`). |
| ③ | **매출·KPI 소스=`/campaign/stats`** | 필드: `applications(신청)·payments(결제)·revenue(매출)·conversionRate(전환율)·contentCount(콘텐츠수)·uploadCount(업로드수)`. KPI 진척바 cur는 **여기서 읽음**. 하드코딩 금지. |
| ④ | **capability 플래그로 demo/real 자동 전환** | 엔진이 `{multiJob15:false, selfHeal:false, bulkKakao:false…}` 제공. UI는 플래그로 real/demo 라벨·동작 분기. **하드코딩 금지.** |
| ⑤ | **기존 자동발송(홍보문자 예약·결제 페이백)=대표 승인 운영** | 별개 레이어로 인지. 시뮬의 회원용 단체카톡 반자동과 **충돌·중복 트리거 금지.** |

### B-4. gAns 구조 (엔진 입력 계약)
```json
{ "want":"...", "pain":"...", "need":"...", "free":"...", "edit":"",
  "diag":{ "types":["유입(고객 없음)","전환(구매 안 됨)"], "sev":"높음", "urg":"급함", "tried":"블로그·지인소개" } }
```
**통합 payload:** `{ "selectedJobs":[0,5], "idx":0, "sAns":{...}, "gAns":{...} }`

### B-5. 진단 → 3대 KPI (KPIPROG/K3)
- `[데모]` idx0: 월 신규 7/10명·월 소득 6200/10000만원·월 계약 6/10건 / generic: 신규발굴 12/20·전환·성사 5/10·관계·재방문 18/30 / 공통 🤝관계개선도 64% / 기간탭 일·주·월·분기·반기·년(기본 월).
- ⚠️ **현재 diag는 수집만, K3는 idx 하드코딩.** diag가 KPI를 아직 구동하지 않음.
- `[도킹]` 온보딩 절대축="처음 3대 KPI 확정→모든 에이전트가 그 KPI 향해 작동, 매 브리핑 최종목표 언급으로 시작."
  - **cur 소스 = `/campaign/stats`** (PART A-③). 매핑: 강의신청←applications / 결제·매출←payments·revenue / 전환율←conversionRate / 활동량←contentCount·uploadCount. **하드코딩 금지.**
  - target=want+워룸 확정. `diag.types`→KPI 가중(유입→발굴/전환→성사/재구매·반복→관계/가격→소득/방향→워룸재정의). `sev·urg`→브리핑·핫리드 우선순위. `tried`→중복제안 회피.
  - 반환: `{ kpis3:[{label,cur,target,unit}x3], relScore:number }`

## PART D — 명세 #3 · 이벤트/화면 계약 (applyEvent payload)

자율루프 = 이벤트 시퀀스. `applyEvent(d, ev)`가 각 이벤트를 화면에 렌더.

| `ev.t` | `[데모]` payload (index 기반) | `[도킹]` 엔진 폴링 payload | UI 동작 |
|---|---|---|---|
|`phase`|`{t:'phase',v,cyc}` v∈상태읽기/계획/실행/확인/반복|동일|루프 단계 표시|
|`convo`|`{t:'convo',i}`→ws.convo[i]|`{t:'convo',id,data:{speaker,icon,text,badge}}`|워룸/대화 추가|
|`kpi`|`{t:'kpi',i}`→ws.kpis[i]|`{t:'kpi',id,data:{value,label,hot?}}`|KPI 카드 갱신|
|`feed`|`{t:'feed',i}`→ws.feed[i]|`{t:'feed',id,data:{t,m,tool,time}}`|밤사이 피드 추가|
|`task`|`{t:'task',i}`→ws.tasks[i]|`{t:'task',id,data:{badge,tt,plan,why,tech,pri}}`|**UI가 task 소유**(A-①). 카드 추가→승인 대기|
|`recsch`|`{t:'recsch'}`|`{t:'recsch',data:{recs,schedule}}`|추천·일정|
|`fail`|`{t:'fail'}`|`{t:'fail',id,data:{reason,options}}`|실패·통제권(failAct)|
|`report`|`{t:'report'}`|`{t:'report',id,data}`|완성 리포트|

- **폴링:** `GET /dashboard/all`(또는 `/events`) → `[{t,id,data}]`. UI가 순차 `applyEvent`. **WebSocket 안 만듦.**
- **변환:** 엔진 응답→UI 형식 변환은 **UI 책임**.

## PART E — 명세 #4 · capability 플래그 목록

`GET /capabilities` → 플래그 객체. UI는 플래그로 real/demo 자동 분기(하드코딩 금지). `true`→실연결+라벨제거 / `false`→데모 동작+"🔎 데모" 라벨+외부행동 비활성.
```json
{
  "chat": true, "approve": true, "campaignStats": true,
  "multiJob15": false,   // 15직업 실프로필 제공 여부 (false=UI 데모 프로필)
  "diary": false,        // 하루일기→모닝브리핑 실연결
  "hotLead": false,      // 핫리드 실시간
  "publish": false,      // 콘텐츠 발행
  "review": false,       // 후기→SNS 홍보
  "bulkKakao": false,    // 단체카톡(반자동) 실연결
  "selfHeal": false,     // 자가수리(ClaudeCode 샌드박스)
  "claim": false,        // 보험금 청구
  "analysis": false,     // 보험 분석
  "newsletter": false,   // 정기 뉴스레터
  "seed": false, "copilot": false, "guard": false,
  "lifecycle": false, "attribution": false, "market": false
}
```

|---|---|
|`campaignStats`|KPIPROG(3대 KPI cur)|
|`diary`|DIARY/모닝브리핑|
|`hotLead`|핫리드·감시병 Hooks|
|`publish`|발행·콘텐츠×채널|
|`review`|REVIEW(후기→홍보)|
|`bulkKakao`|BULK(단체카톡·김영희)|
|`selfHeal`|SELFHEAL(자가수리)|

## PART F — 명세 #5 · 신박기능·자가수리·단체카톡 동작 명세

각 데모: **입력 → 단계 → 출력**. `[도킹]` 시 capability=true면 해당 엔드포인트로 실연결, 아니면 데모.

| 기능(함수) | 입력 | 단계(순차 애니) | 출력/핵심 | 플래그 |
|---|---|---|---|---|

### G-1. 엔드포인트 (실측 반영 최종)

### G-2. 시뮬레이터를 엄마에게 붙이는 방법 (파일 첨부 안 될 때)
전임(이 세션)은 `완성.html`을 outputs에 만들지만 repo에 직접 push는 못 합니다. 대표님이 다리 역할로 아래 중 하나:
1. **GitHub 커밋 (권장)** — `완성.html`을 다운로드 → 엄마 작업 repo(예: `financial-house-building`의 `develop` 또는 `/docs/`)에 추가/커밋. **Claude Code(엄마)는 repo 파일을 직접 읽습니다.** 가장 깔끔.
2. **코드 붙여넣기** — 단일 HTML이라 내용 전체를 엄마 대화창에 붙여넣으면 받습니다(크지만 가능). 또는 "이 내용으로 `docs/지니야빌더_시뮬.html` 만들어줘" 지시.
3. **raw URL / Gist** — GitHub Gist나 raw.githubusercontent.com에 올려 URL 전달 → 엄마가 fetch.
4. **배포 URL** — Vercel(`financial-house-building-dev.vercel.app`)에 올려 URL 공유(화면 확인용).
- ⭐ **이 명세 문서만으로 엄마는 구현 가능합니다**(시뮬=참조용). 실물이 필요하면 1번이 최선.
