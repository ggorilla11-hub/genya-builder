## 2. "실행 페이로드" 저장 방식 · A/B/C

**질문:** 승인 시 무엇을 실행할지(대상·메시지·채널)를 어떻게 담을까?

| 안 | 방식 | 장점 | 단점 |
|---|---|---|---|
| A | 요약 + 대상 전체를 JSON 스냅샷 | 승인 시점=생성 시점 재현 정확 | 연락처 중복 저장(개인정보 확산·명세서 8-2 역행) |
| **B ⭐** | 기준(필터)+템플릿만 저장, **승인 시 시트 재조회** | 개인정보 최소(8-2) · 항상 **최신**(그새 만기 지난 고객 자동 제외) | 생성↔승인 사이 명단 바뀌면 대상수 변동 |
| C | 전체 대상 스냅샷(연락처 포함) | 완전 재현 | A와 같은 개인정보 부담 |

**팀장 추천 ⭐ = B (기준+템플릿 저장, 승인 시 재조회)**
- 근거: 명세서 8-2 "개인정보 최소". 모닝브리핑→몇 시간 뒤 승인 시 **최신 명단** 반영이 더 정확.
- 안전장치: **승인 직전 대상수 재확인**(대량 10건+ 이중확인, 명세서 8-3) → 변동 리스크 커버.

## 3. 발송 실행 채널 범위 · A/B/C

**질문:** 승인 후 "실제 발송"을 지금 어디까지?

| 안 | 범위 | 장점 | 단점 |
|---|---|---|---|
| **A ⭐** | 지금은 **SMS(솔라피)+Gmail** 실동작 / 알림톡은 "심사 후"로 정직 표시 | 시나리오 7 **실발송 완주** 가능(이미 통합됨) · 회장님 매일 임팩트 | 알림톡은 Step 2-D 후 |
| B | 알림톡까지 전부 배선 | 심사 나면 즉시 | 알림톡 템플릿 심사 대기라 지금 실발송 불가(반쪽) |
| C | 발송은 mock(로그만) | 안전 | 시나리오 7 "실제 발송" 완주 안 됨 |

**팀장 추천 ⭐ = A** — 실행되는 채널(SMS/Gmail)로 승인→발송→기록을 **진짜로** 증명. 알림톡은 Step 2-D에서 매핑(경계 명확).

## 4. 서브태스크 → API 6개 (엄마1 통합 계약)

| # | 서브태스크 | 엔드포인트(안) | 비고 |
|---|---|---|---|
| 1 | 결재 시트 스키마 | `결재함` 탭 자동 생성 | id·생성일시·요청내용·대상수·승인상태·결과 (+채널·기준JSON·실행결과) |
| 2 | 지니야 자동 결재 생성 | `POST /api/approval/create` | 기준+템플릿+채널 저장, 상태=대기 |
| 3 | 승인·거부·수정 | `POST /api/approval/act` | action=approve/reject/edit |
| 4 | 결재함 조회 | `GET /api/approval/list` | **엄마1 UI가 부르는 것** — 응답 shape 계약 |
| 5 | 승인 후 실제 실행 | (act=approve 내부에서 재조회→발송) | 대량 이중확인 |
| 6 | 결과 시트 자동 기록 | 실행 후 `결과`·`승인상태=완료` 갱신 | 성공/실패 카운트 |

- **엄마1 통합점(중요):** `GET /api/approval/list` 응답 형태를 엄마1 Mock과 **맞춰야** 교체가 매끄럽습니다.
  → 착수 전 엄마1의 Mock JSON 구조 1개만 공유받으면 그대로 맞추겠습니다(없으면 제가 표준안 제시).
