## 1. 스키마 감지 방식 · A/B/C

**질문:** "이 시트의 컬럼(이름·전화·주소…)을 지니야가 어떻게 알아볼까?"

| 안 | 방식 | 장점 | 단점 |
|---|---|---|---|
| **A ⭐** | 첫 행(헤더) 자동 인식 | 설정 0 · 대표는 기존 시트만 있으면 됨 · 명세서 B-2("컬럼 자동 감지")와 일치 | 헤더가 없거나 지저분하면 약함 |
| B | 사전 스키마 정의(관리자 UI) | 정확·타입지정 가능 | UI 필요(엄마1 의존) · 비개발자 대표에게 설정 부담 |
| C | 하이브리드(자동+수동 오버라이드) | 최종 지향점 | 지금 만들면 오버엔지니어링 |

**팀장 추천 ⭐ = A (첫 행 자동 인식)**
- 근거: 데모 명단이 이미 한글 헤더(고객명·만기일·보험사·직업·비고…)를 갖고 있어 **바로 작동**. 명세서 B-2도 "자동 감지"로 명시.
- 보강 1가지: 말과 컬럼이 다를 때를 위한 **가벼운 동의어 매핑**(예: "주소"↔거주지, "연락처"↔전화·휴대폰). 사전 15~20개만 내장 → 자연어 매칭 정확도↑.
- 미래 경로: 나중에 필요하면 A 위에 오버라이드만 얹어 C로 확장 가능(**지금 A를 골라도 재작업 없음**).

## 2. Function Calling · 도구 정의 · A/B/C

**질문:** "지니야에게 시트 조작 도구(손)를 몇 개나 쥐여줄까?"

| 안 | 구성 | 장점 | 단점 |
|---|---|---|---|
| A | read/update/create/delete 4개 | 단순 | **search 없음** → "홍길동 정보 알려줘"에서 행 못 찾음 |
| B | 세밀 10+개(search·get·update_field·batch_update…) | 정밀 | 프롬프트 비대·오호출↑·오버엔지니어링 |
| **C ⭐** | 중간 5개(search·read·create·update·delete) | 시나리오 딱 맞음 · 명확 | — |

**팀장 추천 ⭐ = C (5개 도구)**
- 근거: 명세서 시나리오가 **검색을 필수로 요구**함 —
  "홍길동 정보 알려줘"(search→read), "이번 주 만기 12명"(search 필터). A는 search가 없어 시나리오 미달.
- 5개 정의:
  - `search_rows` — 조건으로 여러 행 찾기(만기·이름·상태 필터)
  - `read_row` — 한 명 상세 전체
  - `create_row` — 신규 1행
  - `update_row` — 한 행의 특정 필드 수정
  - `delete_row` — 한 행 삭제(강력 승인)
- B의 batch는 Claude가 도구를 여러 번 호출해 대체 → 별도 도구 불필요.

## 3. 승인 게이트 · A/B/C

**질문:** "시트를 바꾸기 전에 대표 승인을 어디까지 받을까?"

| 안 | 정책 | 장점 | 단점 |
|---|---|---|---|
| **A ⭐** | 모든 쓰기 승인 필수(미리보기→승인) | 실수 0 · 신뢰↑ · 명세서 시나리오와 일치 · 구현 단순(게이트 1개) | 잦은 수정 시 클릭 늘어남 |
| B | update·create 자동 / delete·batch만 승인 | 편함 | 조용한 자동 쓰기 = 고객 실데이터에 위험 |
| C | 사용자 설정(기본 모두 승인) | 유연 | 설정 UI 복잡(엄마1 의존) |

**팀장 추천 ⭐ = A (모든 쓰기 승인)**
- 근거: 명세서 시나리오 4(UPDATE)도 **"미리보기 → 승인?"** 를 거침. 8-3 원칙 "실수 방지·미리보기 필수". 실고객 데이터라 초기엔 전부 게이트가 정직.
- delete·대량(10건+)은 **이중 확인**(한 번 더 되묻기) — 명세서 8-3 "대량 발송 이중 확인" 정신.
- 미래: 신뢰 쌓이면 C로 완화 가능(지금 A로 시작해도 재작업 없음).

## 4. 승인 흐름 · 기술 설계 (서버 저장 0 유지)

**중요 제약:** 지니야빌더 대원칙 = **서버에 고객 데이터 저장 0**. 그래서 "미리보기와 승인 사이"에 대기 중인 작업을 서버 DB에 쌓지 않습니다.

**방식(무상태 서명 토큰):**
1. `plan` 단계 — 자연어 → 실행할 CRUD 작업(대상 행·바뀔 값)을 계산해서 **미리보기 + 서명(HMAC-SHA256)** 만 반환. (HMAC은 이미 솔라피에서 쓰는 스택)
2. 대표가 승인하면 클라이언트가 그 작업+서명을 **그대로 되돌려줌**.
3. `commit` 단계 — 서버는 서명만 검증하고 실행. **고객 데이터를 서버가 보관하지 않음**(무상태·단기 만료).

→ 제로 인그레스 원칙과 승인 게이트를 동시에 지킴.
